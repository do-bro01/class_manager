# 기능 테스트 체크리스트

> 작성일: 2026-04-30
> 검증 환경: 본인 명의 Supabase 프로젝트 + Vercel 배포 (Phase 3에서 채움)
> 베이스 코드 + 추가 기능(출결) 모두 포함

테스트는 두 종류로 분리한다.
- **수동 시나리오 테스트**: 사용자 입장에서 흐름이 작동하는지
- **회귀/엣지 테스트**: 잘못된 입력, 권한 위반, 빈 상태 등

각 항목 옆 `[ ]`는 본인이 직접 돌려본 뒤 채울 칸이다. 결과 칼럼에는 `OK` 또는 `발견 이슈 + 처리`를 적는다.

## 8. RLS 직접 검증 (Supabase SQL Editor에서)

다음 쿼리를 강사/학생 각자의 JWT로 실행해 RLS가 의도대로 동작하는지 확인.

```sql
-- 학생 JWT에서: 다른 수업의 attendance_session 조회
select * from attendance_sessions where class_id = '<다른수업id>';
-- 기대: 0 rows

-- 학생 JWT에서: 본인 user_id가 아닌 record 조회
select * from attendance_records where user_id != auth.uid();
-- 기대: 0 rows

-- 학생 JWT에서: check_in_open=false인 session에 직접 INSERT
insert into attendance_records (session_id, user_id, status)
values ('<closed session id>', auth.uid(), 'present');
-- 기대: RLS violation
```

| # | 검증 | 결과 |
|---|---|---|
| R1 | 다른 수업 session 조회 차단 | [ ] |
| R2 | 다른 학생 record 조회 차단 | [ ] |
| R3 | 닫힌 session에 self insert 차단 | [ ] |
| R4 | 강사가 본인 수업 record UPDATE 가능 | [ ] |
