## 0. 회고를 4축으로 나눈 이유

과제 가이드는 회고를 세 축(기능적/기술적/사용성)으로 요청했지만, 평가 비중에서 "AI 활용 능력 25%"가 가장 컸기 때문에 **AI 협업 회고**를 별도 축으로 더했다. 이쪽이 다른 축들과 강하게 얽혀 있어서 분리 안 하면 흐려진다.

| 스펙 항목 | 구현 여부 | 비고 |

| 강사/수강생 역할 분리 | ✅ | `app_metadata.role` 클레임 |

| 수강생 활성/비활성 | ✅ | |
| 공지 작성 | ✅ | 외부 링크 포함 |
| QnA (질문/답변) | ✅ | "답변 완료" 배지 |
| 강사/수강생 대시보드 | ✅ | |
| **출결 (확장 기능)** | ✅ | **스펙에서 우선순위 낮음으로 빠진 걸 추가** |
| 출결 — 자가 체크인 | ✅ | 회차별 토글 |
| 출결 — 출석률 통계 | ✅ | 학생: 본인 / 강사: 회차별 카운트 |

### 스펙에서 의도적으로 빼고 간 것

- **이메일 알림** — 스펙 단계에서 이미 "발송 안 함"으로 결정됨
- **첨부파일 (Supabase Storage)** — MVP 범위 외
- **회차별 출석 인증 시간 제한** — 회차의 `check_in_open` 토글로 대체 (강사가 수동으로 닫음)

### 추가하지 않은 이유가 분명한 것 (협업 로그 §6에서 거절한 제안들)

GeoLocation 검증, audit log, cron 자동 닫기, Realtime — 매력적이지만 1인 강사용 MVP의 복잡도 예산을 넘어선다고 판단해서 거절했다. AI는 기술적으로 가능한 것을 자주 제안하지만, "지금 만들 가치가 있는가"는 항상 사람이 결정해야 한다.

### 평가
- 기획 의도 대비 **빠진 핵심 기능 없음**
- 출결 추가 기능까지 더해 스펙 v1.0 → v1.1로 자연스럽게 확장됨
- 다만 출결 부분의 **수강생 비활성화 시 과거 기록이 강사 출석부에 안 보이는 건 알려진 한계** (active enrollment만 표시하기 때문). 향후 "활성/비활성 모두 보기" 토글을 추가할 여지가 있다.

## 2. 기술적 회고 — 코드를 이해하고 설명할 수 있는가

이 섹션은 면접에서 "이 코드 본인이 짠 거 맞나"를 검증하는 질문에 답할 수 있도록 정리한 것이다. **각 항목을 본인이 직접 한 문단으로 풀어 설명할 수 있어야 한다.**

### 2.1 데이터 흐름

```
[브라우저] ─ form submit ─→ [Server Action] ─→ [Supabase Postgres]
                                  │                     │
                                  │              [RLS 정책 검증]
                                  │                     │
                                  ←─── revalidatePath ──┘
                                  │
[브라우저 갱신: Server Component re-render with fresh data]
```

이 흐름의 핵심은 **Server Action**이라는 새로운 패러다임이다. 기존 REST API에서는 클라이언트가 fetch하고 응답을 받아 상태를 업데이트했지만, Next.js App Router의 Server Action은:

- 서버에서 함수가 직접 실행
- DB 변경 후 `revalidatePath('/dashboard')` 한 줄로 해당 라우트의 캐시 무효화
- React가 알아서 Server Component를 다시 렌더해 새 데이터로 화면 갱신

→ **클라이언트 상태 관리 부담이 크게 줄어든다.** 단, 클라이언트 인터랙션(폼 입력 중 dirty 표시 등)은 여전히 `useState` + `useTransition`으로 다뤄야 한다 — `AttendanceRoster.tsx`가 그 예시.

### 2.2 권한 모델 (이중 방어)

| 계층 | 방법 | 예시 |
|---|---|---|
| 1차: Server Action | `requireInstructorOfClass(classId)` | `attendance.ts`의 거의 모든 함수 |
| 2차: Postgres RLS | 테이블별 정책 | `migration_attendance.sql` |

**왜 이중인가?** Server Action만 있으면 충분해 보이지만, 만약 누군가 Supabase JS SDK를 클라이언트에서 직접 호출해 (anon key로) 테이블을 조회하면 Server Action을 우회할 수 있다. RLS는 그 마지막 방어선이다. 반대로 RLS만 있으면 한국어 에러 메시지를 줄 수 없고, 비즈니스 룰(예: "선택 해제는 DELETE")을 표현하기 어렵다.

### 2.3 RLS 정책의 의미

`attendance_records` 의 INSERT 정책 한 줄을 풀어서:

```sql
with check (
  user_id = auth.uid()                         -- 본인 행만 만들 수 있다
  and exists (                                 -- 그리고 다음 조건이 모두 참이어야 한다:
    select 1
    from attendance_sessions s
    join enrollments e on e.class_id = s.class_id
    where s.id = attendance_records.session_id  -- 같은 session
      and s.check_in_open = true                -- 체크인이 열려있고
      and e.user_id = auth.uid()                -- 본인이
      and e.status = 'active'                   -- 활성 수강생인 수업
  )
)
```

**status 검증을 RLS에 넣지 않은 이유**: RLS의 INSERT 정책은 권한만 검사하고, status가 'present'인지 등 비즈니스 룰은 server action(`selfCheckIn`)에서 강제한다. 이렇게 하면 강사용 정책과 학생용 정책이 충돌하지 않고 정책 의도가 명확해진다. (협업 로그 §2 참조)

### 2.4 출석률 공식

```ts
const denom = counts.present + counts.late + counts.absent
const rate = denom === 0 ? 0 : Math.round(((counts.present + counts.late) / denom) * 100)
```

- 분모에서 `excused`(공결) 제외 — 공결을 분모에 넣으면 학생이 손해
- 분모 0 가드 — 회차가 없거나 모두 공결이면 NaN 방지
- 지각은 출석으로 인정 — 단순화. 지각 0.5점으로 가중치를 줄 수도 있지만 MVP에서는 단순 합산

### 2.5 Next.js 16의 변경점 중 영향 받은 것

- `params: Promise<{...}>` — 동적 라우트 매개변수가 Promise로 바뀜. `const { id } = await params`
- Server Action의 form action 직접 바인딩 + `useTransition` 으로 pending 상태 관리

(자세한 변경점은 작업 시작 전 `node_modules/next/dist/docs/`를 참조하라는 [AGENTS.md](../AGENTS.md) 지시를 따름.)

### 2.6 본인이 답할 수 있어야 하는 질문 모음

면접/시연 시 받을 수 있는 질문들. 각자 답을 준비할 것.

- "왜 강사 역할 판별을 `app_metadata`로 했나? `user_metadata`로 하면 안 되나?"
  → `user_metadata`는 사용자가 클라이언트에서 변경 가능. `app_metadata`는 서버(서비스 키)로만 변경 가능 → 권한 위조 방지.
- "Server Action이면 CSRF 토큰 같은 건 어떻게 처리되나?"
  → Next.js가 자동으로 발급/검증. POST + Origin 헤더 검증 포함.
- "초대 코드 6자가 중복되면?"
  → `invite_code` UNIQUE 제약 + INSERT 시 충돌하면 에러. 현재는 재시도 로직 없음 — 36^6 ≈ 21억 가지라 1인 강사 도구에서는 충돌 확률 무시 가능.
- "출결 회차에 동시에 100명이 자가 체크인 누르면?"
  → `UNIQUE(session_id, user_id)` 제약 + 각 INSERT는 독립 트랜잭션. 같은 사람이 두 번 누르면 두 번째는 거부, 다른 사람들은 모두 성공.

- [ ] 수업 만들고 → 초대 코드 복사 → 학생 등록 → 첫 공지 작성까지 1분 안에?
- [ ] 출석부에서 학생 30명 표시할 때 한 번에 일괄 입력 편한가?
- [ ] 모바일에서 (강의 끝나고 폰으로) 체크인 열기/닫기 가능한가?

### 학생 입장

- [ ] 초대 코드 입력 위치 직관적인가?

### 발견 가능한 사용성 이슈 (본인이 직접 써보고 채울 영역)

- 출석부에서 학생 30명 이상이면 스크롤 길어짐 → 검색/필터 필요할 수 있음
- 회차 카드 리스트가 많아지면 페이징 필요 (현재 무한 스크롤 없음)
- 모바일에서 status 4개 토글 버튼이 한 줄에 안 들어갈 수 있음 → flex-wrap 적용은 했지만 실제 폰에서 확인 필요

### 평가
실사용 강사 한 명(=본인)이 1주 운영해보면 명확해진다. 이 부분은 **테스트 체크리스트와 회고를 분리하지 말고**, 사용해보면서 채워나가는 게 정직하다.

## 4. AI 협업 회고 (가장 중요)

### AI를 어떻게 썼는가

[ai-collaboration-log.md](ai-collaboration-log.md) 가 상세 기록이고, 여기서는 메타 회고를 쓴다.

#### 잘 쓴 패턴

1. **단계 분리**: 기획 → 스펙 → 스키마 → 액션 → UI 순서로 끊어 시켰다. 한 번에 다 시키면 검증이 불가능한 분량이 된다.
2. **AI를 질문하게 강제**: "코드 짜기 전에 결정할 것을 나에게 먼저 물어봐"라는 한 문장이 가장 효과 컸다. 8개의 의사결정이 한 번에 정리됐다.
3. **기존 패턴 참조 강제**: "[기존 NoticeList.tsx] 그대로 따라"라고 하니 AI가 컨벤션 깨는 코드를 만들 여지가 줄었다.
4. **거절을 명시**: "이 제안 거절. 이유는 ..." 을 매번 적었더니 다음 제안에서 같은 실수가 반복되지 않았다.

#### 잘못 쓴 / 위험했던 순간

1. **RLS 정책을 처음엔 그대로 쓸 뻔함**: AI가 만든 정책은 SQL 문법은 맞는데 의미 검증이 안 된 상태였다. 한 줄씩 한국어로 다시 풀어 적어보고 의도와 일치하는지 확인하는 작업이 꼭 필요했다.
2. **타입 추론에 `!` 박으려는 경향**: AI는 non-null assertion을 쉽게 쓴다. 이건 "내가 더 잘 안다"는 선언인데 검증되지 않은 자신감이다. 거의 모든 경우 type narrowing으로 대체할 수 있다.
3. **일어날 수 없는 케이스에 대한 방어 코드**: "오류 처리 다 넣어줘"라고 했더니 절대 발생할 수 없는 경로에까지 try/catch를 둘렀다. 시스템 [지침](../CLAUDE.md)에 "내부 코드 신뢰, 경계에서만 검증"이라고 적혀있는데도 그렇다 — 명시적 거절이 필요했다.

### AI 없이 못 했을 일

솔직히, **혼자였다면 1주일 안에 출결 기능까지는 못 갔을 것**이다. AI가 잘하는 영역:
- 기존 패턴을 빠르게 인식하고 일관되게 복제
- 보일러플레이트(Server Action 시그니처, RLS 정책 뼈대) 즉시 생성
- 타입 정의 / 컴포넌트 props 설계

AI가 잘 못하는 영역 (사람이 결정해야 할 것):
- 비즈니스 룰의 모호한 부분 (예: 출석률 분모에 공결 포함 여부)
- 만들지 말아야 할 기능을 거절하는 결정
- 보안 정책의 의도 검증 — SQL은 맞아도 의미가 틀릴 수 있다
- 사용성 직관 — 실제로 본인이 써봐야 알 수 있는 것

### 한 번 더 한다면 바꿀 점

1. **체크리스트 먼저 작성, 코드는 나중에**: 이번엔 코드 작성 후 테스트 체크리스트를 썼는데, 순서를 뒤집으면 코드 작성 단계에서 엣지 케이스를 놓칠 확률이 줄어든다.
2. **AI 결과물을 git commit 단위로 저장**: 회고 시 어떤 결정이 어디서 들어왔는지 추적 어려움. 작은 단위 커밋이 회고 자료로도 가치 있다.
3. **수동 테스트 기록 자동화**: 시나리오를 머리로 돌리면 까먹는다. 적어도 Playwright 스크립트 한 줄이라도 남기면 다음 회기에 도움이 된다.

## 6. 한 줄 결론

> "AI는 일을 빠르게 만들 수 있지만, **무엇을 만들지 결정하는 것**과 **만든 것의 의미를 검증하는 것**은 여전히 사람의 몫이다. 이번 과제에서 가장 많이 쓴 시간은 코드 작성이 아니라 **AI에게 무엇을 묻고, AI 답변을 어떻게 거절할지**였다."
